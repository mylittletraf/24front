# Фронтенд-задача: рекомендации (блок «Рекомендованное» + шортсы For-You) и обучение профиля

> Хендофф для фронта. Бэкенд реализовал каскадный рекомендательный движок и **персистентный
> обучаемый профиль пользователя** (лайки/дизлайки по актрисам, тегам, атрибутам актрис). Базовые
> конвенции (язык `?lang`, fallback, action-by-uuid, абсолютные URL, JWT) — те же, что в
> `docs/FRONTEND_SPEC.md §0`. Шортс-плеер и базовая лента описаны в `docs/SHORTS_FRONTEND_SPEC.md` —
> этот документ **дополняет** их частью про рекомендации и обучение.

Ничего ломающего в существующих полях нет. Появилось **одно новое поле запроса** (`event` в прогрессе)
и уточнилось поведение двух ранжированных лент.

---

## 0. 🔴 Главное: без `event` в прогрессе рекомендации не учатся

Профиль вкусов пользователя обучается **только** из вызовов прогресса просмотра. Эндпоинт прогресса
принял новое опциональное поле `event`.

### `POST /api/v1/videos/{uuid}/progress/`  (нужен JWT)

| Поле | Тип | Описание |
|---|---|---|
| `watch_progress` | float `0..1` | Доля просмотра (как и раньше) |
| `last_position_seconds` | int ≥ 0 | Позиция в секундах (как и раньше) |
| `event` | `heartbeat` \| `ended` \| `skipped` | **НОВОЕ.** По умолчанию `heartbeat`. Опционально. |

Семантика `event` (что делает бэк):

- **`ended`** — ролик досмотрен до конца/естественно завершился → бэку это **положительный** сигнал:
  начисляет **+вес** актрисам, тегам и атрибутам актрис этого видео в профиль пользователя.
- **`skipped`** — пользователь **быстро смахнул / переключил**, не досмотрев → при низком прогрессе бэк
  начисляет **−вес** (учится реже показывать похожее).
- **`heartbeat`** (default) — обычные апдейты позиции во время проигрывания. Можно слать как раньше;
  полный досмотр (`watch_progress ≥ 0.9`) heartbeat’ом тоже даёт положительный сигнал.

Правила для фронта:

1. **На каждый уходящий ролик слать один `progress` c финальным `watch_progress` и правильным `event`:**
   `ended`, если долистал/досмотрел; `skipped`, если смахнул рано.
2. **Порог «быстрого скипа» вычисляет бэк** (`watch_progress ≤ 0.2` при `event=skipped`). Фронту считать
   ничего не надо — честно передавайте `watch_progress` и сам факт скипа.
3. Достаточно **одного** `ended`/`skipped` на видео — повторные **идемпотентны** (лишний вред нулевой, но
   и смысла слать несколько нет).
4. **Только для авторизованных** (`IsAuthenticated`). Гостям слать не нужно — профиль для них не ведётся.

```jsonc
// Пример: пользователь досмотрел шортс до конца
POST /api/v1/videos/9ea29202-…/progress/
Authorization: Bearer <access>
{ "watch_progress": 0.98, "last_position_seconds": 41, "event": "ended" }
// 200 → { "watch_progress": 0.98, "last_position_seconds": 41 }

// Пример: пользователь смахнул почти сразу
{ "watch_progress": 0.05, "last_position_seconds": 2, "event": "skipped" }
```

> Петля замыкается так: досмотр/скип → `progress` c `event` → обновление профиля → следующий батч
> `recommended` / `shorts/for-you` уже учитывает свежие предпочтения и дизлайки.

## 1. Блок «Рекомендованное» — `GET /api/v1/videos/recommended/`

Горизонтальные (не вертикальные) видео, ранжированные движком. Персонализируется для авторизованного,
**не кешируется**.

| Параметр | Тип | Описание |
|---|---|---|
| `lang` | `ru\|en` | Язык контента |
| `page` | int ≥ 1 | Опционально. Постраничная выдача поверх кэш-снапшота (как в §2). Блоку 5/20 обычно не нужна. |
| `exclude` | uuid через запятую | (без `page`) Уже показанные видео — исключить (механизм «показать ещё») |
| `page_size` | int | По умолчанию 24, максимум 100 (для блока задавать 5 на десктопе / 20 на мобильном) |

- **Блок у плеера (5 десктоп / 20 моб) — одноразовый запрос**, пагинация не нужна: просто `?page_size=5|20`,
  один вызов, рисуем ряд. `?page=`/`?exclude` подключать только если появится отдельная **страница
  «Рекомендации»** с догрузкой.
- **Ответ (батч-режим):** `{ "results": [...] }` — **нет** `count`/`next`/курсора. Карточка = каталожная +
  блок `actors`/`categories`/`tags` (см. §4), detail-запрос за ними не нужен. При `?page=` ответ дополняется
  `page`/`page_size`/`has_more` (как в §2).
- **`seed` здесь не читается** (в отличие от шортсов) — это лента «для вас вообще», не «похоже на X».
- **Персонализация:** для авторизованного слать `Authorization: Bearer` — тогда работают профиль,
  атрибуты и дизлайки. Для гостя — трендовая выдача (валидно).

Что реализовать:

1. **Бесконечная подгрузка через `?exclude`** (курсора нет): хранить set показанных `uuid`, добавлять их
   в `exclude` при каждом до-запросе.
2. **Ограничить длину `exclude`** — последние ~150–200 uuid. Уже просмотренные бэк и так исключает по
   истории, так что «потерять» ничего не страшно, а querystring не должен расти бесконечно.
3. Дедуп по `uuid` на клиенте (подстраховка; сервер тоже исключает).
4. Не кешировать ответ авторизованного пользователя на клиенте надолго — каждый вызов это свежий ре-ранкинг.

```jsonc
GET /api/v1/videos/recommended/?lang=ru&page_size=24
// → { "results": [ {VideoCard}, … ] }

// «Показать ещё»: добавляем показанные uuid
GET /api/v1/videos/recommended/?lang=ru&exclude=uuid1,uuid2,uuid3
```

## 2. Шортсы «Для вас» — `GET /api/v1/videos/shorts/for-you/`

Ранжированная вертикальная лента, построенная **от текущего открытого ролика**. Это отдельная механика от
базовой `/videos/shorts/` (см. `SHORTS_FRONTEND_SPEC.md`).

| Параметр | Тип | Описание |
|---|---|---|
| `lang` | `ru\|en` | Язык контента |
| `seed` | uuid | **Текущий открытый шортс.** Смещает выдачу в его сторону, сам в неё не попадает |
| `page` | int ≥ 1 | **(рекомендуется для бесконечного скролла)** Опционально. Номер страницы поверх кэш-снапшота ранга. |
| `exclude` | uuid через запятую | (legacy-режим, без `page`) Уже показанные — исключить |
| `page_size` | int | По умолчанию 24, максимум 100 |
| `categories`/`include_tags`/`actors`/… | slug через запятую | Полный `VideoFilter` (скоуп ленты) |

- **Два режима догрузки:**
  - **`?page=N` (рекомендуется для свайпа)** — стабильная постраничная выдача поверх **кэшированного
    снапшота** ранга. Ранг считается **один раз** на `(user, seed, фильтры)` и кэшируется (~5 мин), поэтому
    страницы **не пересчитывают** движок и **не прыгают**. Ответ: `{ "results": [...], "page", "page_size",
    "has_more" }`. Листать, пока `has_more=true`. `?exclude` в этом режиме **не нужен**.
  - **Батч + `?exclude` (legacy)** — без `?page`: `{ "results": [...] }`, клиент копит показанные uuid в
    `?exclude`. Оставлен для совместимости; для бесконечного свайпа предпочитайте `?page`.
- **Снапшот учитывает просмотренное на момент создания** (~5-мин окно). Свежие просмотры внутри сессии
  подхватятся после истечения TTL — для свайп-ленты это незаметно.
- Батч-режим сам по себе **не кешируется** (персонализирован); `?page`-режим переиспользует снапшот.

Что реализовать:

1. **При открытии шортса передавать его `uuid` как `seed`** — лента строится «похоже на то, что смотрит».
2. **Бесконечный скролл через `?page=N`** (предпочтительно): `page=1,2,3…` с фиксированным `page_size` и
   `seed`. Листать, пока `has_more=true`. Растущий `?exclude` больше не нужен — снапшот сам ведёт
   последовательность. (Legacy-режим `?exclude` остаётся доступен, если удобнее.)
3. **Пре-фетч следующей страницы заранее** (например, когда осталось 3–4 ролика до конца текущей) — для
   бесшовного свайпа.
4. **Порядок стабилен между страницами** (кэш-снапшот) — можно держать загруженные страницы в памяти сессии.
5. Связать с §0: на каждый свайп слать `progress` c `event` → после истечения TTL снапшота следующая лента
   уже учитывает свежие предпочтения/дизлайки.

```jsonc
GET /api/v1/videos/shorts/for-you/?lang=ru&seed=9ea29202-…&page_size=24
// → { "results": [ {VideoShort}, … ] }

// Догрузка
GET /api/v1/videos/shorts/for-you/?lang=ru&seed=9ea29202-…&exclude=uuidA,uuidB
```

> Разница с базовой `GET /videos/shorts/`: та — **курсорная** лента по популярности (`{next, results}` или
> `?page=N` с `count`), для бесконечного свайпа «просто популярное». `for-you` — ранжированный батч без
> курсора, «похоже на текущее». Не смешивать пейджеры.

## 3. 🔴 Плейбек HLS — обязательный отдельный запрос (напоминание)

В карточке/шортсе поле `sources.hls` **намеренно `null`** (мастер-плейлист подписанный и короткоживущий,
payload кешируется). Для реального воспроизведения:

1. В момент старта проигрывания дёрнуть `GET /api/v1/videos/{uuid}/playback/`, HLS-URL брать оттуда
   (сегменты — `/videos/{uuid}/hls/…`). Подробности — `SHORTS_FRONTEND_SPEC.md`.
2. Не кешировать HLS-URL надолго — протухает.
3. `poster`/`trailer` из карточки — для превью/тизера без запроса плейбека.

## 4. Кросс-срезовые моменты

- **Карточки «Рекомендованного» и шортса теперь содержат `actors`/`categories`/`tags`** — оверлей/подписи
  рисуются без detail-запроса. Формат полей одинаковый: `actors` = `[{uuid,name,slug,gender}]`,
  `categories`/`tags` = `[{uuid,name,slug}]` (локализованные; `tags` включают собственные теги видео + «общие
  теги» актрис). Атрибуты актрис (цвет волос и т.п.) и `studios` в карточку **не** входят — при необходимости
  брать из `GET /videos/<uuid>/` (detail).
- **Обычные каталожные карточки** (`GET /videos/`, `trending`/`popular`/`new`) — по-прежнему **без**
  `actors`/`tags` (namely `VideoCard`): они лёгкие и кэшируются. Обогащены только `recommended` и шортсы.
- **Пустая выдача**: при исчерпании пула (всё в `exclude`/просмотрено) `recommended`/`for-you` вернут
  `{"results": []}` — нужен UX «вы всё посмотрели» / сброс `exclude` / переключение на трендовое.
- **Вёрстка вертикали** — по флагу `is_vertical`. Явных `width/height` в payload нет.
- **Явная кнопка «не интересно/скрыть»** (если планируется): сейчас негативный сигнал в профиль приходит
  **только** через `event=skipped`. Существующий `like/dislike` пишет счётчики, а не профиль. Отдельный
  эндпоинт «дизлайк-в-профиль» — **доработка бэка** (пока нет). Нужен — заводите задачу на бэк.
