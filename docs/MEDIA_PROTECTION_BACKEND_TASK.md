# Backend task: защита медиа (signed URLs + nginx secure_link)

Задача для **бэкенда + storage(nginx)**. Цель — сделать так, чтобы медиа (видео/HLS, превью) нельзя
было массово хотлинкать/качать «в обход», и чтобы прямые постоянные ссылки на storage не утекали
навсегда. Подписывание делает **бэк**, проверку — **storage(nginx)**. Фронт только запрашивает
свежую ссылку и отдаёт её плееру; подписывать на фронте нельзя (секрет утечёт в браузер).

Эта задача — предусловие для перевода фронта на прямые signed-URL. Пока бэк отдаёт постоянные
storage-URL, фронт проксирует их через свой `/media/*` (времянка) — см. `src/app/media/[...path]/route.ts`.

## Что нужно сделать

### 1. UUID вместо последовательных id
Публичный идентификатор видео — `uuid` (он уже есть в API). В путях storage не использовать
числовой `id` (перебираемый): путь к файлам — по `uuid`, напр. `/v/<uuid>/720/seg_000.m4s`.

### 2. Эндпоинт playback (контракт для фронта)
```
GET /api/v1/videos/{uuid}/playback/
```
- **Адресация по `uuid`** (это action, а не публичное чтение по slug).
- Бэк проверяет: доступность видео, при необходимости — авторизацию/гео/лимиты/реферер/UA/сессию.
- Возвращает **short-TTL signed** ссылку на мастер-плейлист (+ опционально превью/постер):

```json
{
  "hls": "https://storage-7.example.com/v/<uuid>/master.m3u8?expires=1719999999&token=abc123",
  "expires_at": "2026-06-21T18:42:00Z",
  "poster": "https://storage-7.example.com/v/<uuid>/poster.webp?expires=...&token=..."
}
```

Требования к ответу:
- Заголовок **`Cache-Control: no-store`** (ссылка короткоживущая, кешировать нельзя).
- TTL подписи — **1–5 минут** (`expires`).
- Если видео недоступно — `403`; если ссылку дёрнули после протухания, storage отдаёт `403/410`
  (фронт по `410`/`403` перезапросит playback).

> Важно про кеш фронта: текущий `GET /videos/{slug}/` кешируется на фронте через ISR
> (`revalidate=60`). **Поэтому signed-URL нельзя класть в этот ответ** — он закешируется и протухнет.
> Нужен отдельный `/playback/` (no-store), который фронт дёргает по «Play» каждый раз свежим.

### 3. Подпись и проверка (HMAC)
```
token = HMAC_SHA256(secret, path_prefix + "|" + expires [+ "|" + session_id] [+ "|" + ip])
```
- `secret` знают только бэк и storage(nginx), на фронт не попадает.
- Подписывать **префикс директории видео** (`/v/<uuid>/`), а не каждый файл по отдельности —
  тогда один токен покрывает мастер-плейлист, варианты (`720.m3u8`) и все сегменты на TTL.
- Опционально привязать к `session_id`/`ip`, чтобы ссылку нельзя было расшарить.

### 4. HLS: сегменты тоже должны быть под защитой
Мастер- и вариант-плейлисты ссылаются на сегменты относительными путями (`seg_1080_000.m4s`),
поэтому плеер **не добавит** `?expires&token` сам. Варианты (выбрать один, на усмотрение storage):
- **A. Переписать URI в плейлистах** — бэк/стораж генерит `.m3u8`, где у каждого сегмента и
  вложенного плейлиста уже есть `?expires=...&token=...` (один и тот же токен на префикс).
- **B. Signed cookie** — при playback ставится подписанная кука на префикс `/v/<uuid>/`, nginx
  валидирует её для всех файлов (учесть, что storage на отдельном домене → `SameSite=None; Secure`).
- **C. `auth_request`/OpenResty(Lua)** — nginx на каждый запрос под `/v/<uuid>/` проверяет
  `token`+`expires` против префикса.

Рекомендация: **A или C** (без кук, проще кросс-доменно). HLS вместо одиночного `mp4`.

### 5. nginx на storage (пример, secure_link)
```nginx
location ~ ^/v/(?<vid>[0-9a-f-]+)/ {
    secure_link        $arg_token,$arg_expires;
    secure_link_md5    "$secret/v/$vid/$arg_expires";   # подпись по префиксу директории

    if ($secure_link = "")  { return 403; }   # подпись не сошлась
    if ($secure_link = "0") { return 410; }   # истёк expires

    add_header Cache-Control "public, max-age=31536000, immutable";  # сегменты иммутабельны
    # ...отдать файл
}
```
(secure_link здесь подписывает префикс `/v/<uuid>/`, поэтому один токен валиден для master/variant/
segments до `expires`. При варианте C — то же самое на Lua с HMAC-SHA256.)

### 6. Вторичные меры
- **Запрет hotlink по `Referer`/`Origin`** на storage/nginx — как дополнительный фильтр (ненадёжно
  само по себе, но отсекает казуальное встраивание).
- **Rate limit** по IP/сессии/пользователю на `/playback/` и на storage.
- **Логи скачиваний** + бан подозрительных IP (аномальный объём/частота).
- `X-Robots-Tag: noindex` на файловые URL storage.

## ⚠️ CORS на сегментах storage (обязательно для HLS.js)

Плеер на сайте (video.js / HLS.js в Chrome/Firefox) грузит сегменты `.ts/.m4s` через `fetch`/XHR,
поэтому **storage(nginx) должен отдавать CORS-заголовки на файлах сегментов**, иначе браузер
заблокирует загрузку и видео не запустится (в Safari нативный HLS работает и без этого — отсюда
«у меня играет, у других нет»).

Сейчас: master/variant-плейлисты (хост API) уже отдают `Access-Control-Allow-Origin: *`, а **сегменты
на storage — нет**. Добавить на storage в `location` сегментов:
```nginx
add_header Access-Control-Allow-Origin "*" always;   # или конкретный origin сайта
add_header Access-Control-Allow-Methods "GET, HEAD, OPTIONS" always;
add_header Access-Control-Expose-Headers "Content-Length, Content-Range, Accept-Ranges" always;
# (Range уже работает: Accept-Ranges: bytes присутствует)
```

## Чего НЕ надо (не защита)
CORS, скрытие API, обфускация JS, запрет правой кнопки, UUID без signed-URL (только от перебора),
только-Referer (обходится).

## Что сделает фронт, когда это будет готово (вне этой задачи)
1. По «Play» дёргает `GET /videos/{uuid}/playback/` (свежий, no-store) и отдаёт `hls` плееру.
2. Переключает видео на **прямые signed-URL**, минуя свой `/media`-прокси → это и анти-лич (TTL),
   и разгрузка трафика фронт-VDS (видео идёт браузер↔storage напрямую).
3. На `403/410` от storage — повторный запрос `/playback/`.
4. `/media`-прокси остаётся максимум для превью-картинок (если они не подписаны) или убирается.

## Acceptance
- `GET /videos/{uuid}/playback/` отдаёт signed HLS с TTL 1–5 мин и `Cache-Control: no-store`.
- Прямой запрос файла storage без валидного `token`/после `expires` → `403/410`.
- Полный прогон HLS (master → variant → segments) играет по одной выданной ссылке, все сегменты
  авторизованы (через A/B/C).
- Ссылку, выданную одному, нельзя переиспользовать после TTL (и, если включено, с другой сессии/IP).
