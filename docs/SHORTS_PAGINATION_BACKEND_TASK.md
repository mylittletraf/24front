# Бэкенд-задача: page-number пагинация для ленты шортсов (`/videos/shorts/`)

> **СТАТУС: ✅ ГОТОВО (вариант 1).** Режим включается наличием `?page=` на `GET /videos/shorts/`
> (action `shorts`). С `?page=N&page_size=` отдаётся DRF-форма `{count, next, previous, results}`;
> без `page` — прежний курсорный ответ `{next, previous, results}` без `count` (полноэкранный свайп
> `ShortsFeed` не меняется). `page_size` по умолчанию 24, максимум 100. Работает со всеми фильтрами
> ленты (`categories`, `include_tags`, `actors`, `lang`); `count` считается по отфильтрованной
> выборке. Порядок `(-trending_score, -id)` детерминирован в пределах одного запроса; для
> авторизованных персонализированная выборка пересчитывается из живых данных, поэтому строки могут
> «дрейфовать» между страницами — как и в каталоге (свойство offset-пагинации).
> Бэкенд: `apps/videos/api/v1/pagination.py` (`VideoPageNumberPagination`) + гейт в
> `VideoViewSet.paginator` (`apps/videos/api/v1/views.py`).
> Фронт: `/shorts` использует нумерованный `PaginationNav` (`getShortsPaged`).

Хендофф для бэкенда. На фронте сделали **страницу-каталог `/shorts`** — сетка вертикальных шортсов
(как каталог видео), клик по карточке открывает полноэкранный плеер (`/shorts/<slug>`). Сейчас на ней
работает **cursor**-пагинация (кнопки «Старше/Новее»), потому что `GET /videos/shorts/` отдаёт
**только cursor** (`next`/`previous`, **без `count`** и без номеров страниц) — `?page=` эхо-возвращается
в `next`, но игнорируется, `previous` всегда `null` (проверено curl). Чтобы включить **нумерованную
пагинацию** (1, 2, 3 … N) как в каталоге видео, бэку нужно отдавать общее число и уметь прыгать на
страницу N.

## Что нужно
Поддержать **page-number режим** на `GET /videos/shorts/`:
- Принимать `?page=N` (1-based) и существующий `?page_size=` (как сейчас).
- В этом режиме отвечать DRF-формой PageNumberPagination:
  ```json
  { "count": 1234, "next": "<url|null>", "previous": "<url|null>", "results": [ … ] }
  ```
  `count` — общее число подходящих вертикальных видео (для расчёта числа страниц).
- Работает со всеми текущими параметрами ленты: `categories`, `include_tags`, `actors`, `lang`.
- Сохранить персонализацию: при наличии JWT порядок учитывает историю просмотров; анонимам — тренды.
  Порядок в пределах одной выборки **детерминированный**, чтобы страница N не «плыла» между запросами.

## Важно про совместимость (cursor оставить по умолчанию)
`/videos/shorts/` сейчас курсорный, и на cursor завязана **полноэкранная лента** (бесконечный свайп
в `ShortsFeed` подгружает страницы по `next`). **Не ломать её.** Режим переключать так же, как
договоримся для `/videos/` (см. `CATALOG_PAGINATION_BACKEND_TASK.md`) — желательно одинаково:
1. Режим включается **наличием `?page=`** (есть `page` → PageNumber; иначе cursor по-умолчанию). ← удобно фронту.
2. Либо явный флаг `?pagination=page`.

## Критерии приёмки
- `GET /videos/shorts/?page=1&page_size=30` → `{count, next, previous, results}`, `count` = всего
  вертикальных видео, `results` = первые 30.
- `GET /videos/shorts/?page=2&page_size=30&categories=molodaia` → второй блок из 30 по фильтру;
  `count` = всего в фильтре; страницы 1 и 2 не пересекаются (стабильный порядок).
- Существующий **курсорный** ответ без `?page` не меняется (полноэкранный свайп работает как раньше).
- Сообщите фронту триггер режима (п.1/п.2) и любые отличия в именах/поведении.

## Что делает фронт после этого (для контекста, не задача бэка)
- Заменяем cursor «Старше/Новее» на `/shorts` на нумерованный пейджер `PaginationNav`
  (`count`/`page_size` → число страниц), как на главной и в `/tag|/category|/studio`.
- `self-canonical` + `noindex,follow` на страницах ≥2 (листинг шортсов не индексируется; индексируемая
  поверхность — `/shorts/<slug>`).
- Файлы фронта: `src/app/shorts/page.tsx` (сейчас `getShortsFeed` + `ListingPagination`),
  `src/lib/api/shorts.ts` (добавим `getShortsPaged` по аналогии с `getVideosPaged`).
