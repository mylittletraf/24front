# Задача (backend): троттл и ротация `/auth/refresh/` (частые 429 / разлогины)

## Симптом
Пользователей часто выкидывает с ошибкой **429 «Слишком много запросов»**. Возникает при обычном
использовании — перезагрузки страницы, несколько вкладок.

## Как устроен фронт (контекст)
Гибридный BFF: `access` живёт в памяти вкладки, `refresh` — в httpOnly-cookie. На **каждую загрузку
страницы** фронт делает «тихий» bootstrap сессии → BFF идёт на бэк **`POST /auth/refresh/`** →
получает новый `access` + **ротированный `refresh`** (cookie перезаписывается) → затем `GET /me/`.

То есть `/auth/refresh/` — **высокочастотный** эндпоинт по дизайну (раз на загрузку, на вкладку),
а не редкий как login.

> Фронт уже оптимизирован: одновременные bootstrap'ы (StrictMode/ремаунты/вкладка) **коалесятся в
> один** запрос к `/auth/refresh/` (см. `src/lib/auth/bff-client.ts`). Это убирает дубли в пределах
> одной вкладки, но не решает кросс-вкладочный и общий троттл — это на стороне бэка.

## Что не так на бэке (гипотезы — проверить)
1. **Слишком строгий троттл на `/auth/refresh/`.** Похоже, refresh попал в один бакет с
   login/anon (например `5/min`). Для эндпоинта, который дёргается на каждую загрузку, это мало →
   нормальные перезагрузки/вкладки дают 429.
2. **Ротация refresh нетолерантна к параллелизму.** Две почти одновременные загрузки (разные
   вкладки) шлют один и тот же refresh; первый ротирует токен, второй приходит с уже использованным
   → если включён reuse-detection, гасится вся сессия → разлогин.
3. **Троттл по IP, а не по пользователю/сессии** → несколько юзеров за одним NAT/прокси делят лимит.

## Что сделать
1. **Отдельный, щедрый троттл для `/auth/refresh/`** (отвязать от login/anon-бакета). Ориентир:
   десятки запросов в минуту на пользователя/сессию — refresh не должен 429-иться при обычной
   навигации. Брутфорс-защиту оставить на `/auth/login/`.
2. **Grace-window на ротацию refresh:** в течение N секунд (например 10–30с) принимать
   **предыдущий** refresh-токен (или один и тот же токен повторно) и возвращать актуальную пару,
   чтобы параллельные bootstrap'ы из разных вкладок не валили сессию. Reuse-detection не должен
   срабатывать на近-одновременные легитимные повторы.
3. **Троттлить по пользователю/сессии**, где возможно, а не только по IP.
4. На 429 всегда отдавать **`Retry-After`** (сейчас иногда без него — фронт не может показать,
   через сколько повторить).

## Проверка / Acceptance criteria
- [ ] 10–20 перезагрузок страницы подряд и 3–4 открытые вкладки **не** приводят к 429 и не
      разлогинивают.
- [ ] Два почти одновременных `/auth/refresh/` с одним refresh-cookie оба завершаются успешно
      (grace-window), сессия не гасится.
- [ ] `/auth/login/` сохраняет строгий анти-брутфорс троттл (его не трогаем).
- [ ] Ответ 429 содержит заголовок `Retry-After`.

## Диагностика (как воспроизвести)
- В DevTools → Network фильтр по `auth/refresh` и `auth/session`: при перезагрузках смотреть частоту
  и коды (429 / 401).
- Проверить конфиг DRF throttling: scope/rate для refresh vs login; включён ли rotate + reuse-detection.
